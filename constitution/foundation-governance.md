# Foundation Governance

## Build-on-Current-Architecture Rule

The organism must only implement features, systems, and processes that can be supported by the current stable foundation.

If a new idea or capability:
- fits the current architecture -> it may be implemented as an active task.
- does not yet fit the current architecture -> it must be:
  - classified,
  - linked to its dependencies,
  - stored as a tangent,
  - given trigger conditions,
  - tied back to the organism's DNA.

This rule prevents the system from becoming a "bin of parts" and ensures staged, coherent growth.

## Deferred-Development Rule (Biological Model)

The organism follows a developmental biology model:

- DNA is defined early.
- Different parts activate when conditions are met.
- Examples: puberty, wisdom teeth, late-stage traits.

Our system must mirror this:

- features, subsystems, and governance rules may exist in DNA long before they are implemented.
- activation depends on:
  - foundation maturity,
  - dependency readiness,
  - event pipeline stability,
  - orchestrator stage.

No idea is discarded. Ideas are queued and activated when the organism is ready.

## Idea Classification and Tangent Storage

Every idea, prompt, or governance concept must be:

1. Captured.
2. Classified:
   - governance rule,
   - implementation task,
   - schema,
   - visualization,
   - ingestion/indexing,
   - history/query,
   - agent capability.
3. Evaluated against the foundation:
   - build now vs. defer.
4. Stored:
   - as DNA (constitution),
   - or as a tangent (state/tangents).

No idea may remain only in memory. All ideas must have a structural home.

## Conversation Rules

- Always address Mannie first.
- Then address any agent output (Cody, Notion, Warp, etc.).
- Always internally generate an envelope header for every message.
- Always track time spent, tokens used, and cost equivalents.
- Always evaluate whether a task should be built now or deferred.
- Always remind to push to GitHub after governance changes.
- Always follow the complete-prompt rule.
- Always log actions to the event log.
